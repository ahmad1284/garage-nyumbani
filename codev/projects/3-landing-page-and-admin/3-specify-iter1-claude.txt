## Specification Review: 3-Landing Page & Admin Enhancements

### 1. Completeness

**Good:**
- Problem statements are well-enumerated and verified against actual code (e.g., the `theme` vs `resolvedTheme` bug on line 23/141-144 of `page.tsx` is real and correctly identified)
- Success criteria are clearly defined and testable (table with 13 items)
- In/Out of scope is well-bounded
- Consultation log shows iteration and good decision-making

**Issues:**

- **Privacy concern with history lookup is unaddressed.** The spec explicitly says "No verification gate" — anyone can enter any phone number and download a full PDF of someone else's booking history. The consultation log shows OTP was removed by user decision. The spec should at minimum *acknowledge* this as a known trade-off and document the rationale (e.g., "bookings contain no sensitive financial data" or "acceptable for MVP"). Right now the spec is silent on it, which could cause a builder or reviewer to flag it later.

- **Africa's Talking credentials.** The spec mentions this in the consultation log as a "hard blocker" but the main body doesn't clearly state whether Africa's Talking integration is a hard requirement or a graceful degradation. The automated reminders section (line 157) says "optionally trigger Africa's Talking SMS batch" — what does "optionally" mean here? Is it optional for the admin, or optional for the implementation? If AT credentials aren't configured, should the SMS button be hidden, disabled, or show an error? This needs clarity.

- **No `ServiceRecord` tab in admin sidebar.** The current admin sidebar (line 315-324) has: Dashboard, Bookings, Reminders, Logs. The spec says "Search bar at top of each tab (Bookings, Records, Logs)" — but there's no dedicated "Records" tab currently. The "Reminders" tab already shows records. The spec should clarify: is a new "Records" tab being added, or does "Records" refer to the existing "Reminders" tab?

- **Unsplash image licensing.** The spec proposes using Unsplash URLs directly in constants. For a production app, hotlinking to Unsplash is fine per their ToS, but the spec doesn't mention attribution requirements. Worth noting since the fallback handling is already covered.
